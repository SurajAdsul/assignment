Document QA Service

A small HTTP API that answers questions over a fixed, in-memory knowledge base using retrieval-augmented generation — with citations, a confidence label, and a hard rule that it never asserts anything the documents don't say.

✓ Hallucination is structurally impossible

The four seed documents (refund policy, API rate limits, data retention, authentication) are embedded in the service, so it is fully self-contained: no network, no API keys, no external services. Ask a question, get back the supporting source sentences with their document citations — or an honest refusal when the knowledge base can't answer.

🎯 Grounded

Answers are stitched verbatim from retrieved sentences. There is no LLM to invent claims.

🛡️ Safe by default

Below a calibrated score floor, the service refuses instead of guessing.

🔍 Observable

Every response exposes the candidate chunks and their scores for debugging.

⚡ Self-contained

Hand-rolled TF-IDF, zero ML dependencies, instant startup.

How this project was built The design was stress-tested decision-by-decision before any code was written (a structured "grilling" session). The resulting domain glossary lives in CONTEXT.md and the one architectural decision worth preserving is recorded in docs/adr/0001. This page is the human-readable tour of all of it.

⚖️ citations vs used_chunks — read this

These are intentionally different — the spec's field name ("used") versus its description ("retrieved") forced the question, and we resolved it deliberately:

  • citations is the grounding signal — documents that cleared the threshold and actually back the answer. This is what you show users.
  • used_chunks is the observability signal — every candidate retrieval surfaced, with its score, including rejected ones. Above, doc_b is a candidate (0.347) but is not cited — it's only a spurious word collision ("request a refund" vs "API requests").

DesignArchitecture

A request flows through four stages. Each is a small, independently testable module.

1 · Chunking

Each document is split into one chunk per sentence. The seed docs are written one fact per line, so tight chunks make for precise answers and fine-grained scoring. There are 4 documents → 12 chunks.

2 · Retrieval

A hand-rolled TF-IDF with cosine similarity. Tokens are lowercased, stopword-filtered, and lightly stemmed (so refund/refunds match). Each chunk is indexed on its sentence plus its document title, so a topical question matches even when it shares no words with the exact answering sentence. It lives behind a Retriever protocol, so swapping in embeddings or a vector DB touches nothing else.

3 · Grounding

This stage owns the anti-hallucination policy via two thresholds with two distinct jobs (see Calibration for why two):

ThresholdValueJob
TAU0.15Refusal floor. If the best chunk scores below it → refuse honestly.
RELATIVE_CUTOFF0.6Citation gate. A supporting chunk joins the answer only if it scores ≥ 60% of the best chunk.

top_k is a ceiling, not a quota: asking for 3 chunks never forces 3 weak sentences into the answer.

4 · Answering

The answer is extractive: it is the retrieved source sentences, each tagged with [doc_id]. There is no language model, so the answer cannot contain a claim that isn't in the knowledge base.

Confidence heuristic

LabelWhen
lowRefusal — nothing cleared TAU.
mediumAnswered, but the top score is weak (< TAU_HIGH = 0.40) or the chunks cover < 50% of the question's content terms.
highStrong top score and good question coverage.

The coverage term keeps it honest on partially-answerable questions: ask something with five specific terms where only one matches the corpus, and you get medium, not a confident high.

DesignADR & The Threshold Refinement

ADR 0001 — Grounding over fluency

Decision

We answer by stitching together verbatim sentences retrieved from the knowledge base (each tagged with its doc_id) rather than generating prose with an LLM, and we refuse whenever no retrieved chunk clears the score threshold.

Why it's recorded: a reviewer of a "RAG service" expects an LLM call and will ask "where's the model?" The explicit no-LLM-fallback stance matters most — it stops a future contributor from "improving" the service by bolting on a general-knowledge fallback, which would silently violate the core requirement that answers come only from the knowledge base.

Consequence: answers read as terse quoted sentences, not conversational prose. That is the intended trade-off. If an LLM is ever added, it must be constrained to the retrieved chunks and verified against them — and this ADR superseded.

The two-threshold refinement

The original plan used a single absolute threshold τ to gate both the refusal decision and which chunks get cited. Calibration against real scores proved this can't work — and surfacing that contradiction is exactly what the design-review process is for.

⚠️ Why one threshold failed

As a refusal floor, τ must be low (~0.15): the weakest legitimate question tops out at 0.456, so a high floor would refuse real questions. But as a citation gate, τ must be higher: a spurious cross-topic match scored 0.347, which sits below genuine corroborating second chunks (0.44–0.48 in other queries). No single absolute number separates them.

The fix keeps the absolute floor for refusal and gates supporting chunks relative to the best chunk — a supporting chunk joins only if it scores ≥ 60% of the top score. That adapts per-query: it kills the over-citation (0.347 / 0.634 = 0.55 → dropped) while keeping real corroboration (0.483 / 0.525 = 0.92 → kept). This is a tuning refinement, not architecture, so it lives in code comments and the README rather than a formal ADR.

retrieval.py — normalization & TF-IDF

The conservative stemmer bridges the common morphology in this corpus without pulling in a real NLP library:

app/retrieval.pypython
def stem(token: str) -> str:
    if len(token) > 4 and token.endswith("ies"):
        return token[:-3] + "y"
    if len(token) > 4 and token.endswith("ing"):
        return token[:-3]
    if len(token) > 4 and token.endswith("ed"):
        return token[:-2]
    if len(token) > 3 and token.endswith("s") and not token.endswith("ss"):
        return token[:-1]   # "refunds" → "refund", but "access" stays "access"
    return token

Vectors are sparse dicts. IDF is smoothed (log((n+1)/(df+1)) + 1), vectors are L2-normalized, and out-of-vocabulary query terms are dropped so an unknown word never tanks the cosine score. Similarity is just the dot product of two normalized vectors.

answer.py — the grounding core

This is the heart of the safeguard. Refuse first; otherwise the best chunk always grounds the answer and supporting chunks must clear the relative cutoff:

app/answer.pypython
retrieved = retriever.search(question, top_k)
used_chunks = [_serialize(sc) for sc in retrieved]   # all candidates, always

if not retrieved or retrieved[0].score < tau:
    return AnswerResult(REFUSAL_TEXT, citations=[], confidence="low", used_chunks=used_chunks)

# Best chunk always grounds; supporting chunks must be within RELATIVE_CUTOFF.
top_score = retrieved[0].score
cutoff = max(tau, RELATIVE_CUTOFF * top_score)
grounding = [sc for sc in retrieved if sc.score >= cutoff]

answer = " ".join(f"{sc.chunk.text} [{sc.chunk.doc_id}]" for sc in grounding)

Confidence is then a pure function of the top score and question coverage — note that low is reserved for refusals, so the lowest answered tier is medium:

app/answer.pypython
def _confidence(top_score: float, coverage: float) -> str:
    if top_score >= TAU_HIGH:
        return "high" if coverage >= COVERAGE_MIN else "medium"
    return "medium"

models.py — input validation

Pydantic does validation declaratively, so bad input returns 422 with zero hand-written code and the contract self-documents in OpenAPI:

app/models.pypython
class AnswerRequest(BaseModel):
    question: str = Field(..., min_length=1)
    top_k: int = Field(2, ge=1, le=3)        # out of range → 422

    @field_validator("question")
    @classmethod
    def question_not_blank(cls, value: str) -> str:
        if not value.strip():
            raise ValueError("question must not be blank")
        return value

main.py — the endpoint

The FastAPI app builds the retriever once at startup, maps the domain result to the response schema, and emits a structured, no-PII log line per request (top_score, grounded, confidence, latency_ms) — the raw signals you'd ship to metrics in production.

That clean separation is what makes the refusal safeguard reliable: "What is the capital of France?" shares no content words with any chunk, scores exactly 0.0, and is refused without ambiguity.

🔬 Calibration caught a real bug

Probing with top_k=2 revealed over-citation: "How long do I have to request a refund?" cited doc_b too, because "request" collides with "API requests". The data showed the spurious chunk scored 0.347 — below genuine corroboration in other queries — which is precisely why the relative cutoff replaced a second absolute threshold. (Full reasoning in the ADR section.)

Deep diveTesting

27 tests across four files run in ~0.03s. They cover the three required cases plus the contract guarantees and a regression test for the over-citation fix.

test_api.py

End-to-end via FastAPI's TestClient: grounded answer, refusal, invalid input (×5), top_k ceiling, citation/used_chunk shape, no-over-citation.

test_answer.py

Confidence bands, coverage demotion, the relative-cutoff gate, and refusal behaviour at the domain layer.

test_retrieval.py

Stemming, stopword removal, correct top-document ranking, zero-score for out-of-domain, determinism.

test_chunking.py

Sentence splitting (12 chunks), unique namespaced chunk ids, title carry-through.

The three explicitly required cases — a grounded answer, an unanswerable question, and invalid input — are each covered, the last by five distinct rejection tests (too-large top_k, zero, blank question, missing question, wrong type).

ReferenceTradeoffs & Limitations

  • Extractive, not generative. Answers are stitched source sentences, not fluent prose — a deliberate trade for a guaranteed no-hallucination property and zero external dependencies.
  • Lexical retrieval misses synonyms. "How long until my data is purged?" won't match "retained"/"removal" well. Embeddings would fix this at the cost of a heavy dependency — not worth it at this scale.
  • The stemmer is crude. It bridges plurals and common suffixes but will miss pairs like delete/deletion. Conservative by design.
  • TF-IDF is hand-rolled, not scikit-learn. Keeps the footprint tiny and the algorithm legible; behind the Retriever seam it's a drop-in swap.
  • Thresholds tuned on a tiny sample. Defensible for a prototype; production would calibrate on logged real queries.

ReferenceIf This Went to Production

🔎 Retrieval quality

Hybrid search — combine dense embeddings with the lexical score (reciprocal-rank fusion); a real lemmatizer; query expansion; re-rank top candidates.

📊 Observability

Ship the existing log signals as metrics: refusal rate, confidence distribution, and score histograms are the early warnings that the corpus or thresholds have drifted. Trace IDs; sampled, consented QA logging for offline eval.

✅ Evaluation

A labelled question→expected-doc set in CI to catch retrieval regressions; track grounding precision/recall as the corpus changes.

🔒 Hardening

Auth, rate limiting, request size limits, and per-tenant knowledge bases.
